# Workflow сайт Мишаня

## Контекст работы

Над сайтом работают минимум два человека на разных компьютерах. Поэтому локальная версия сайта у Сергея, версия второго участника и версия на GitHub могут отличаться.

Рабочая папка сайта:

`C:\Users\User\Desktop\Cloude CODE\1. Продажи Мишаня\New SITE`

GitHub-репозиторий:

`https://github.com/royaleventil-code/mishanya-site.git`

## Правило перед началом работы

Если Сергей пишет только «продолжаем работать по сайту» или похожую фразу без конкретной задачи, стартовое действие одно:

1. Открыть локальный сайт `http://localhost:3003/`.
   Порт `3003` закреплен за этим проектом, чтобы не мешать параллельной работе CRM на `3000`.
2. Ничего не придумывать и не искать, где остановились.
3. Ждать конкретную задачу от Сергея.

Перед реальными изменениями сайта нужно свериться с GitHub, если Сергей не сказал иначе:

1. Проверить текущую ветку и локальные изменения.
2. Проверить, есть ли на GitHub версия новее локальной.
3. Если на GitHub есть новые изменения, сначала подтянуть их.
4. Только после этого начинать править сайт.

## Если локальная версия новее GitHub

Если после проверки видно, что локальная версия опережает GitHub, ассистент не пушит сам молча.

Сначала нужно спросить Сергея:

`Локальная версия сайта новее GitHub. Залить её на GitHub?`

Только после подтверждения Сергея можно делать commit/push.

## Если есть локальные незакоммиченные изменения

Если в рабочей папке есть незакоммиченные изменения:

1. Не затирать их.
2. Не делать reset/checkout без прямой просьбы Сергея.
3. Сначала коротко сообщить, какие файлы изменены.
4. Если pull может конфликтовать с локальными изменениями, остановиться и спросить Сергея, что делаем: сохранить, закоммитить, отложить или разобрать вручную.

## Локальная работа в течение дня

Сергей сам управляет последовательностью задач по сайту. Ассистент не выбирает следующий кусок работы самостоятельно, не отмечает страницы/программы как готовые без прямой команды и не делает выводы вида «очевидно, следующий шаг такой-то».

Сайт сначала правим локально.

Порядок:

1. Запустить локальный сервер сайта на порту `3003`, если он не запущен.
2. Открыть локальный сайт `http://localhost:3003/`.
3. Вносить только те правки, которые Сергей прямо попросил.
4. После каждой правки не прогонять `lint`, `build`, тесты и другие проверки автоматически.
5. После каждой правки не кликать по сайту в браузере и не проверять весь сценарий автоматически.
6. Проверки, прогоны и браузерные клики делать только по прямой просьбе Сергея или перед push, если Сергей подтвердил такую проверку.
7. В течение рабочего процесса не пушить каждую мелкую правку на GitHub.
8. В конце рабочего дня собрать готовые изменения, коротко показать Сергею итог и спросить подтверждение на push.

GitHub используем как общую точку синхронизации, а не как место для каждой промежуточной правки.

## Базовый порядок работы

1. Открыть папку сайта.
2. Проверить GitHub и локальное состояние.
3. Подтянуть GitHub, если там версия новее.
4. Если локальная версия новее GitHub, спросить Сергея про загрузку на GitHub.
5. После синхронизации выполнять задачу по сайту.
6. После изменений проверить сайт локально.
7. Перед отправкой на GitHub показать Сергею кратко, что изменилось, и спросить подтверждение на push.

## Короткое правило

Сайт не правим вслепую и не ведём самостоятельно. На старте открыть `http://localhost:3003/` и ждать задачу. Правки только по команде Сергея. Проверки и браузерные клики после правок — только если Сергей попросил.
